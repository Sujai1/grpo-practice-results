GRPO Practice — Can a 135M Model Learn Math via RL?

What I learned trying to run GRPO on GSM8K with a model that's probably too small. Personal notes, instructional tone.

W&B project → 2026-04-16 · SmolLM2-135M · GSM8K · 3× g6.xlarge on-demand

Experimental setup

  • Model: SmolLM2-135M base → SFT-Alpaca (LoRA r=16, lr=1e-3, Alpaca 2K) → merged as standalone model.
  • Task: GSM8K math word problems (train split: 973 easy problems with ≤50-token gold solutions).
  • Reward: format reward (+0.1 for emitting #### <num>) + correctness reward (+1.0 for matching gold answer).
  • Infra: 3× g6.xlarge (L4, 24GB VRAM), on-demand. TRL 1.1.0, bf16.

Three-stage pipeline

The full pipeline was: SFT-Alpaca → CoT warm-start → STaR bootstrap → GRPO. Each stage builds on the previous checkpoint. The idea is to give GRPO the best possible starting policy so it has enough correct-sample density for advantage estimation to work.

Approaches — what we tried to make it work

  1. Format + correctness hybrid reward — format reward saved us from zero gradient early on, but correctness never climbed. The format signal (+0.1) gave the model something to latch onto during the first few steps, preventing a total cold-start failure.
  2. Heavy CoT warm-start (3 epochs on gold) — pushed pass rate from ~1% to 2.3%. This was the single most effective intervention. Training directly on gold chain-of-thought solutions gave the model the best chance of producing correct answers during GRPO rollouts.
  3. Tighter curriculum (≤50-token solutions only) — reduced the training set to 973 single-step problems. The idea was to remove multi-step problems the model had zero chance of solving, concentrating training signal on tractable examples.
  4. Higher temperature (T=1.5) — more exploration but not enough underlying capability. Higher temperature increases diversity in rollouts, which should help GRPO find more correct samples per group. But if the model can't solve the problem at any temperature, diversity doesn't help.
  5. STaR bootstrap (3 rounds, K=64) — found ~55 correct samples per round but pass rate didn't improve across rounds (2.25% → 1.83% → 1.75%). The SFT on noisy model-generated samples actually hurt format knowledge. Each round overwrote some of the clean formatting learned during the CoT warm-start.

Key learnings

Minimum capability threshold

RL needs a minimum capability threshold. GRPO's advantage computation requires mixed-reward groups (some correct, some wrong). At 2% pass rate, 85–95% of groups have zero reward variance → zero gradient. The model can't learn what it can't occasionally do.

STaR can backfire. SFT'ing on the model's own noisy correct completions overwrote format knowledge from the warm-start. Pass rate didn't improve across rounds and format compliance dropped from 50% to <1%.

Format reward is a clever hack but not enough. The model learned to emit #### <num> in the first smoke test (50% format rate), but this signal plateau'd immediately — it's a binary feature the model picks up fast, then provides no gradient.

CoT-SFT (supervised on gold) dominated because it doesn't require the model to discover correct answers on its own — it provides them directly. For a model this small, the gap between "can imitate a solution" and "can discover a solution" is enormous.

The KL reference trick matters. With LoRA, disabling the adapter reveals the merged SFT-Alpaca model as reference. We specifically merged the SFT adapter so GRPO regularizes toward SFT behavior, not raw pre-training. This keeps the policy from drifting into gibberish during training, even when reward signal is sparse.

Troubleshooting a failing GRPO run — what the metrics told us

When GRPO isn’t learning, the W&B dashboard has five diagnostic signals. Here’s exactly what we saw in each, what it meant, and what the research literature says to do about it.

1. frac_reward_zero_std: 85–95%

What it means: 85–95% of prompt groups had identical reward for all 16 completions (all wrong). GRPO computes advantage = (reward − group_mean) / group_std. When std=0, advantage=0, gradient=0. We were wasting 85–95% of our compute on prompts that produced zero learning signal.

Fix: difficulty-aware prompt sampling

Only train on prompts where the model has a 10–50% pass rate — the “Goldilocks zone” where groups will have mixed rewards. Maintain a running per-prompt pass-rate estimate from recent rollouts, and sample prompts weighted by p×(1−p) to maximize advantage variance. This is what DAPO (Dynamic Advantage Policy Optimization) does. A simpler variant: use a global running reward mean instead of the group mean, so even all-wrong groups get a small negative advantage (“you’re worse than average”) rather than zero.

2. rewards/correctness_reward/mean: flat at 0%

What it means: The model almost never produced a correct final answer during training rollouts. The reward landscape is essentially flat — there’s no hill for policy gradient to climb. Format reward provided a tiny nonzero signal (~0.1%), but it plateau’d immediately because it’s a binary feature the model picked up in the first few steps.

Fix: Process Reward Models (dense reward)

Replace binary final-answer reward with step-level scoring. A PRM evaluates each reasoning step: “48/2 = 24” gets credit even if the final answer is wrong. This converts a sparse reward problem into a dense one. Three implementation options ranked by cost:

  1. Regex arithmetic checker (~30 lines): parse every “A op B = C” pattern, verify C is correct. Give +0.05 per correct operation. Cheap and directly applicable to GSM8K.
  2. Math-Shepherd style: generate 256 completions per prompt, label each step by whether it correlates with reaching the correct final answer. Train a small step-level classifier.
  3. LLM-as-judge: have a larger model score each step. Expensive but zero-training.

3. completions/clipped_ratio: 0.85–0.96

What it means: 85–96% of completions hit the maximum length (200 tokens) without naturally terminating. The model was rambling — generating text until truncation rather than producing a concise answer with #### and stopping. This happened because the STaR bootstrap overwrote the format-following behavior from the warm-start: format compliance dropped from 50% (pre-STaR) to <1% (post-STaR).

Lesson: STaR’s SFT rounds need quality filtering, not just correctness filtering. A completion that accidentally gets the right number but lacks #### formatting should be rejected. Without this, each STaR round erodes format knowledge, creating longer and messier generations that are harder for the next round to learn from. This is a compounding failure mode.

4. kl: ~0.0007 (near zero)

What it means: The policy barely moved from the reference. KL divergence near zero means the LoRA adapter learned essentially nothing — gradients were too small to push the policy away from the starting point. This is consistent with the zero-variance-group diagnosis: no gradient → no policy update → no KL divergence.

In a healthy GRPO run, KL should rise to 0.1–1.0 as the policy learns. If KL stays near zero, the optimization is dead. If KL shoots past 2.0, the policy is diverging (lower β or LR). We were stuck at the “dead” end.

5. grad_norm: 0.00007 (six orders of magnitude below normal)

What it means: Gradients were essentially zero. Normal GRPO grad norms are 0.01–1.0. Ours were 10,000× smaller. This is the downstream consequence of zero-variance groups: no advantage → no policy gradient → no parameter updates. The model was doing expensive forward and backward passes that produced nothing.

The Dr. GRPO fix: better advantage computation

Even with all the above fixes, standard GRPO has a subtle bias in its advantage computation. When a group has 1/16 correct, the correct completion gets advantage ≈ +3.9 while the 15 wrong ones get ≈ −0.26. This asymmetry means the gradient is dominated by a single lucky sample that may have been correct by accident.

Dr. GRPO (Liu et al., 2025) fixes this by subtracting the expected reward under the reference policy instead of the group mean, and normalizing per-token instead of per-sequence. The code change is ~20 lines in the advantage computation. TRL 1.1.0 doesn’t implement this by default, but you can subclass GRPOTrainer and override _compute_advantages.

Our run hit the first four failure modes simultaneously — all symptoms of the same root cause: the model is below the capability threshold for RL on this task. The fixes above (PRM, difficulty filtering, Dr. GRPO) would each help, but the most honest fix is a bigger model.

What I'd do differently

  1. Start with Qwen-0.5B or larger. 135M likely lacks the representational capacity for multi-step arithmetic regardless of training. The minimum viable model size for GSM8K is probably 500M+.
  2. Try a simpler task first (2-digit addition). Validate the GRPO pipeline separately from model capability. If GRPO can't teach addition to a 135M model, the pipeline is broken. If it can, the problem is GSM8K difficulty.
  3. Use a Process Reward Model for denser signal. Binary final-answer checking gives one bit of reward per rollout. A PRM scores each reasoning step, providing gradient even when the final answer is wrong.
  4. Run more STaR rounds with quality filtering. Reject samples that don't follow the format, not just incorrect ones. The format regression from STaR was avoidable with a stricter filter.
  5. Consider knowledge distillation. Have a larger model generate solutions, SFT on those, then GRPO. This gives the small model a better starting distribution without requiring it to discover solutions from scratch.
